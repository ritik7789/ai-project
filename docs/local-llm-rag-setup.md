# Local LLM Setup for RAG (Ollama: Llama3/Qwen3)

This project can now use a local LLM for final RAG answers while still using
our retrieval layer for source selection.

## 1) Prerequisites

- Backend dependencies installed from `requirements.txt`
- Ollama running locally
- One of these models available:
  - `llama3:8b`
  - `qwen3:8b`

Quick checks:

```bash
ollama list
curl http://127.0.0.1:11434/api/tags
```

## 2) Environment variables

Set these before starting FastAPI.

```bash
export RAG_LLM_ENABLED=true
export RAG_LLM_BASE_URL=http://127.0.0.1:11434
export RAG_LLM_MODEL=llama3:8b
export RAG_LLM_TIMEOUT_SECONDS=60
export RAG_LLM_TEMPERATURE=0.1
```

For Qwen instead:

```bash
export RAG_LLM_MODEL=qwen3:8b
```

## 3) Run backend

```bash
uvicorn app.main:app --reload
```

## 4) Validate quickly

```bash
curl -X POST http://127.0.0.1:8000/rag/query \
  -H "Content-Type: application/json" \
  -d '{"question": "How is payroll processed?"}'
```

Expected behavior:
- `sources` still comes from retrieval.
- `answer` is generated by LLM when enabled/reachable.
- `llm_used: true` when LLM was used.
- If LLM is down/unreachable, service falls back to retrieval chunk answer.

## 5) Troubleshooting

- `llm_used` is always `false`:
  - Ensure `RAG_LLM_ENABLED=true`
  - Ensure Ollama is running and reachable at `RAG_LLM_BASE_URL`
- Slow responses:
  - Increase `RAG_LLM_TIMEOUT_SECONDS`
  - Try a smaller/faster model if available
- Vague answers:
  - Improve ERP document quality/chunking
  - Keep `RAG_LLM_TEMPERATURE` low (`0.0` to `0.2`)
